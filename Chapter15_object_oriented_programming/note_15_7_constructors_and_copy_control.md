<a id="id1"></a>
## ✅ 知识点1: 继承层次中的拷贝控制基础

**理论**
* 与任何其他类一样，继承层次中的类控制其类型对象被**创建**、**拷贝**、**移动**、**赋值**或**销毁**时发生什么：
  * 如果类（基类或派生类）没有自己定义拷贝控制操作之一，**编译器将合成该操作**
  * 与往常一样，这些成员的合成版本可能是**删除的函数**

---

<a id="id2"></a>
## ✅ 知识点2: 虚析构函数的必要性

**理论**
* **继承对基类拷贝控制的主要直接影响**：基类通常应定义**虚析构函数**
* 析构函数需要是虚的，以允许继承层次中的对象被**动态分配**
* 当我们`delete`指向动态分配对象的指针时，会运行析构函数
* 如果该指针指向继承层次中的类型，指针的**静态类型**可能与被销毁对象的**动态类型**不同
* **示例场景**：
  1. 如果我们`delete`一个`Quote*`类型的指针，该指针可能指向`Bulk_quote`对象。
  2. 如果指针指向`Bulk_quote`对象，编译器需要知道它应该运行`Bulk_quote`析构函数

**教材示例代码**
```cpp
class Quote {
public:
    // virtual destructor needed if a base pointer pointing to a derived object is deleted
    virtual ~Quote() = default;  // dynamic binding for the destructor
};
```

---

<a id="id3"></a>
## ✅ 知识点3: 虚析构函数的动态绑定行为

**理论**
* 与其他虚函数一样，析构函数的虚特性被**继承**
* 因此，从`Quote`派生的类具有虚析构函数，无论它们使用合成的析构函数还是定义自己的版本
* **关键保证**：只要基类析构函数是虚的，当我们`delete`指向基类的指针时，**将运行正确的析构函数**

**教材示例代码**
```cpp
Quote *itemP = new Quote;      // same static and dynamic type
delete itemP;                  // destructor for Quote called
itemP = new Bulk_quote;        // static and dynamic types differ
delete itemP;                  // destructor for Bulk_quote called
```

---

<a id="id4"></a>
## ✅ 知识点4: 虚析构函数的继承特性

**理论**
* **虚特性的继承**：与其他任何虚函数一样，析构函数的虚特性被继承
* 从`Quote`派生的类具有虚析构函数，无论它们使用合成的析构函数还是定义自己的版本
* 只要基类析构函数是虚的，当删除指向基类的指针时，将运行正确的析构函数

---

<a id="id5"></a>
## ✅ 知识点5: 删除指向派生类对象的基类指针时的未定义行为警告

**注意点**
* ⚠️ **警告**：如果基类析构函数不是虚的，**对指向派生类对象的基类指针执行`delete`具有未定义行为**

---

<a id="id6"></a>
## ✅ 知识点6: 虚析构函数与三/五法则的例外

**理论**
* **重要例外**：基类析构函数是"如果类需要析构函数，则它也需要拷贝和赋值"这一经验法则的重要例外
* **基类几乎总是需要析构函数，以便使析构函数成为虚的**
* 如果基类为了使其成为虚的而具有空析构函数，那么**该类具有析构函数并不表示也需要赋值运算符或拷贝构造函数**

---

<a id="id7"></a>
## ✅ 知识点7: 虚析构函数关闭合成移动操作

**理论**
* **基类需要虚析构函数这一事实对基类和派生类的定义有重要间接影响**：如果一个类定义了析构函数（即使它使用`= default`使用合成版本），**编译器将不会为该类合成移动操作**
* 如我们所见，`Quote`没有合成移动操作，因为它定义了析构函数
* 当我们移动`Quote`对象时，将使用（合成的）拷贝操作
* 正如我们将要看到的，`Quote`没有移动操作这一事实意味着其派生类也没有

---

<a id="id8"></a>
## ✅ 知识点8: 合成的拷贝控制成员在继承中的执行方式

**理论**
* 基类或派生类中的合成拷贝控制成员的执行方式与其他合成的构造函数、赋值运算符或析构函数相同：
  * 它们**逐成员**地初始化、赋值或销毁类本身的成员
* 此外，这些合成成员使用基类的相应操作来逐成员地初始化、赋值或销毁对象的**直接基类部分**
* **构造函数执行顺序示例**：
  * 合成的`Bulk_quote`默认构造函数运行`Disc_Quote`默认构造函数，后者又运行`Quote`默认构造函数
  * `Quote`默认构造函数默认初始化`bookNo`成员为空字符串，并使用类内初始化器将`price`初始化为零
  * 当`Quote`构造函数完成时，`Disc_Quote`构造函数继续，使用类内初始化器初始化`qty`和`discount`
  * 当`Disc_Quote`构造函数完成时，`Bulk_quote`构造函数继续但没有其他工作要做
* 类似地，合成的`Bulk_quote`拷贝构造函数使用（合成的）`Disc_Quote`拷贝构造函数，后者使用（合成的）`Quote`拷贝构造函数
* `Quote`拷贝构造函数拷贝`bookNo`和`price`成员；`Disc_Quote`拷贝构造函数拷贝`qty`和`discount`成员

**注意点**
* 值得指出的是，**基类成员本身是合成的**（如我们的`Quote`层次结构中的情况）还是具有用户提供的定义并不重要。
* **重要的是相应的成员是可访问的**且**不是删除的函数**

---

<a id="id9"></a>
## ✅ 知识点9: 合成拷贝控制对基类成员的依赖条件

**理论**
* 每个`Quote`类都使用合成的析构函数
* 派生类隐式地使用，而`Quote`类通过将其（虚）析构函数定义为`= default`显式地这样做
* 合成的析构函数（像往常一样）为空，其隐式销毁部分销毁类的成员（§13.1.3, p. 501）
* 除了销毁自己的成员外，派生类中的析构函数的销毁阶段还销毁其**直接基类(direct base)**
* 该析构函数又调用其自己的直接基类的析构函数（如果有的话），依此类推，直到层次结构的根

---

<a id="id10"></a>
## ✅ 知识点10: 基类与派生类中的删除拷贝控制(Base Classes and Deleted Copy Control in the Derived)

**理论**
* 基类或派生类中的合成默认构造函数或任何拷贝控制成员可能因与任何其他类相同的原因被定义为**删除的(deleted)**（§13.1.6, p. 508, 和 §13.6.2, p. 537）
* 此外，定义基类的方式可能导致派生类成员被定义为删除的：
  * 如果基类中的默认构造函数、拷贝构造函数、拷贝赋值运算符或析构函数被删除或不可访问（§15.5, p. 612），则派生类中的相应成员被定义为删除的，因为编译器无法使用基类成员来构造、赋值或销毁对象的基类部分
  * 如果基类具有不可访问或删除的析构函数，则派生类中的合成默认构造函数和拷贝构造函数被定义为删除的，因为无法销毁派生对象的基类部分
  * 与往常一样，编译器不会合成删除的移动操作。如果我们使用`= default`请求移动操作，如果基类中的相应操作被删除或不可访问，则它在派生类中将是删除的函数，因为基类部分无法移动。如果基类析构函数被删除或不可访问，移动构造函数也将被删除

**教材示例代码**
```cpp
class B {
public:
    B();
    B(const B&) = delete;
    // other members, not including a move constructor
};

class D : public B {
    // no constructors
};

D d;          // ok: D's synthesized default constructor uses B's default constructor
D d2(d);      // error: D's synthesized copy constructor is deleted
D d3(std::move(d));  // error: implicitly uses D's deleted copy constructor
```

<a id="id11"></a>
## ✅ 知识点11: 移动操作与继承(Move Operations and Inheritance)

**理论**
* 正如我们所见，**大多数基类定义虚析构函数**。因此，默认情况下，**基类通常不会获得合成的移动操作**
* 此外，默认情况下，从**没有移动操作的基类派生的类也不会获得合成的移动操作**
* 因为基类中缺少移动操作会抑制其派生类的合成移动，所以**如果合理，基类通常应定义移动操作**
* 我们的`Quote`类可以使用合成版本。但是，`Quote`必须显式地定义这些成员
* 一旦它定义了移动操作，它还必须显式定义拷贝版本（§13.6.2, p. 539）

**教材示例代码**
```cpp
class Quote {
public:
    Quote() = default;                    // memberwise default initialize
    Quote(const Quote&) = default;        // memberwise copy
    Quote(Quote&&) = default;             // memberwise copy
    Quote& operator=(const Quote&) = default;  // copy assign
    Quote& operator=(Quote&&) = default;       // move assign
    virtual ~Quote() = default;
    // other members as before
};
```

**注意点**
* 现在，`Quote`对象将被逐成员地拷贝、移动、赋值和销毁
* 此外，从`Quote`派生的类也将自动获得合成的移动操作，除非它们有以其他方式阻止移动的成员

<a id="id12"></a>
## ✅ 知识点12: 派生类拷贝控制成员的基础原则

**理论**
* 如我们在§15.2.2（第598页）中所见，派生类构造函数的初始化阶段初始化派生对象的**基类部分(base-class part(s))**以及初始化其自己的成员
* 因此，派生类的拷贝和移动构造函数必须**拷贝/移动其基类部分的成员以及派生类中的成员**
* 类似地，派生类赋值运算符必须赋值派生对象基类部分中的成员
* 与构造函数和赋值运算符不同，**析构函数只负责销毁派生类分配的资源**
* 回忆一下，对象的成员被隐式销毁（§13.1.3, p. 502）。类似地，派生对象的基类部分也被自动销毁

**注意点**
* ⚠️ **警告**：当派生类定义拷贝或移动操作时，该操作负责拷贝或移动**整个对象**，包括**基类成员(base-class members)**

---

<a id="id13"></a>
## ✅ 知识点13: 定义派生类拷贝或移动构造函数

**理论**
* 当我们为派生类定义拷贝或移动构造函数时（§13.1.1, p. 496, 和 §13.6.2, p. 534），我们通常使用相应的基类构造函数来初始化对象的基类部分

**教材示例代码**
```cpp
class Base { /* ... */ };
class D: public Base {
public:
    // by default, the base class default constructor initializes the base part of an object
    // to use the copy or move constructor, we must explicitly call that
    // constructor in the constructor initializer list
    D(const D& d): Base(d)          // copy the base members
        /* initializers for members of D */ { /* ... */ }
    D(D&& d): Base(std::move(d))    // move the base members
        /* initializers for members of D */ { /* ... */ }
};
```

**注意点**
* 初始化器`Base(d)`将`D`对象传递给基类构造函数
* 虽然原则上`Base`可以具有`D`类型参数的构造函数，但在实践中这非常不可能
* 相反，`Base(d)`将（通常）匹配`Base`拷贝构造函数
* `D`对象`d`将被绑定到该构造函数中的`Base&`参数
* `Base`拷贝构造函数将`d`的基类部分拷贝到正在创建的对象中
* 如果省略了基类的初始化器，`Base`默认构造函数将用于初始化`D`对象的基类部分
* 假设`D`的构造函数从`d`拷贝派生成员，这个新构造的对象将被奇怪地配置：**其`Base`成员将保存默认值，而其`D`成员将是来自另一个对象的数据拷贝**
* ⚠️ **警告**：默认情况下，基类默认构造函数初始化派生对象的基类部分。如果我们想拷贝（或移动）基类部分，必须在派生类构造函数的**初始化列表(constructor initializer list)**中显式使用基类的**拷贝（或移动）构造函数**

<a id="id14"></a>
## ✅ 知识点14: 派生类赋值运算符(Derived-Class Assignment Operator)

**理论**
* 与拷贝和移动构造函数一样，派生类赋值运算符（§13.1.2, p. 500, 和 §13.6.2, p. 536）必须**显式赋值其基类部分**

**教材示例代码**
```cpp
// Base::operator=(const Base&) is not invoked automatically
D &D::operator=(const D &rhs)
{
    Base::operator=(rhs);  // assigns the base part
    // assign the members in the derived class, as usual,
    // handling self-assignment and freeing existing resources as appropriate
    return *this;
}
```

**注意点**
* 该运算符首先通过显式调用基类赋值运算符来赋值派生对象的基类部分的成员
* 基类运算符将（可能）正确处理自赋值，并且如果适当，将释放左操作数基类部分中的旧值并从`rhs`赋值新值
* 一旦该运算符完成，我们继续做任何需要做的事来赋值派生类中的成员
* 值得注意的是，**派生类构造函数或赋值运算符可以使用其相应的基类操作，而不管基类是定义了自己的该操作版本还是使用合成版本**
* 例如，对`Base::operator=`的调用执行`Base`类中的拷贝赋值运算符。该运算符是由`Base`类显式定义的还是由编译器合成的并不重要

<a id="id15"></a>
## ✅ 知识点15: 构造函数与析构函数中的虚函数调用限制

**理论**
* 如我们所见，派生对象的基类部分首先被构造
* 当基类构造函数执行时，对象的派生部分是**未初始化的(uninitialized)**
* 类似地，派生对象以相反的顺序销毁，因此当基类析构函数运行时，派生部分已经被销毁
* 因此，**当这些基类成员执行时，对象是不完整的(incomplete)**
* 为了适应这种不完整性，**编译器将对象视为在构造或销毁期间其类型发生变化**
* 也就是说，当对象被构造时，它被**视为与构造函数具有相同的类**；对虚函数的调用将绑定为对象具有与构造函数本身相同的类型
* 类似地，对于析构函数也是如此
* 这种绑定适用于直接调用或从构造函数（或析构函数）调用的函数中间接调用的虚函数
* 为了理解这种行为，考虑如果从基类构造函数调用虚函数的派生类版本会发生什么
* 这个虚函数可能访问派生对象的成员。毕竟，如果虚函数不需要使用派生对象的成员，派生类可能可以使用其基类中的版本
* 然而，当基类构造函数运行时，这些成员是**未初始化的**
* 如果允许这种访问，程序可能会崩溃

**注意点**
* 📋 **术语提醒**：如果构造函数或析构函数调用虚函数，**运行的版本是与构造函数或析构函数本身的类型相对应的那个**

---

<a id="id16"></a>
## ✅ 知识点16: 派生类析构函数(Derived-Class Destructor)

**理论**
* 回忆一下，对象的数据成员在析构函数体完成后被**隐式销毁(implicitly destroyed)**（§13.1.3, p. 502）
* 类似地，对象的基类部分也被隐式销毁
* 因此，与构造函数和赋值运算符不同，**派生类析构函数只负责销毁由派生类分配的资源**

**教材示例代码**
```cpp
class D: public Base {
public:
    // Base::~Base invoked automatically
    ~D() { /* do what it takes to clean up derived members */ }
};
```

**注意点**
* **对象销毁的顺序与构造的顺序相反**：首先运行派生析构函数，然后调用基类析构函数，沿继承层次结构向上回溯

<a id="id17"></a>
## ✅ 知识点17: 继承的构造函数(Inherited Constructors)基础

**理论**
* 在新标准下，派生类可以**重用(reuse)**其直接基类定义的构造函数
* 虽然正如我们将看到的，这种构造函数在通常意义上不是"继承的"，但通常称这种构造函数为"继承的"
* 出于与类只能初始化其直接基类相同的原因，**类只能从其直接基类继承构造函数**
* 类**不能继承默认、拷贝和移动构造函数**
* 如果派生类没有直接定义这些构造函数，编译器将照常合成它们
* 派生类通过提供**using声明(using declaration)**来继承其基类构造函数，该声明命名其（直接）基类

**教材示例代码**
```cpp
class Bulk_quote : public Disc_quote {
public:
    using Disc_quote::Disc_quote;  // inherit Disc_quote's constructors
    double net_price(std::size_t) const;
};
```

**注意点**
* 通常，using声明只是使名称在当前作用域中可见
* 当应用于构造函数时，**using声明导致编译器生成代码**
* 编译器为基类中的每个构造函数生成一个派生类构造函数
* 也就是说，对于基类中的每个构造函数，编译器在派生类中生成一个具有相同参数列表的构造函数

**生成的构造函数形式**
```cpp
derived(parms) : base(args) { }
```
其中`derived`是派生类的名称，`base`是基类的名称，`parms`是构造函数的参数列表，`args`将派生构造函数的参数传递给基类构造函数

**示例**
```cpp
Bulk_quote(const std::string& book, double price,
           std::size_t qty, double disc):
    Disc_quote(book, price, qty, disc) { }
```

* 如果派生类有自己的任何数据成员，这些成员被**默认初始化(default initialized)**（§7.1.4, p. 266）

## ✅ 知识点18: 继承构造函数的特性(Characteristics of an Inherited Constructor)

**理论**
* **访问级别不变**：与普通成员的using声明不同，构造函数的using声明**不改变继承构造函数的访问级别**
* 例如，无论using声明出现在哪里，基类中的私有构造函数在派生类中是私有构造函数；类似地，受保护和公共构造函数也是如此
* **explicit和constexpr继承**：
  * using声明**不能指定explicit或constexpr**
  * 如果基类中的构造函数是**explicit**（§7.5.4, p. 296）或**constexpr**（§7.5.6, p. 299），则继承的构造函数具有相同的属性
* **默认参数的处理**：
  * 如果基类构造函数有默认参数（§6.5.1, p. 236），**这些参数不会被继承**
  * 相反，派生类获得**多个继承构造函数**，其中每个具有默认参数的参数被连续省略
  * 例如，如果基类有一个具有两个参数的构造函数，其中第二个具有默认值，派生类将获得两个构造函数：一个具有两个参数（没有默认参数），第二个具有单个参数（对应于基类中最左边非默认参数）
* **两个例外**（如果基类有几个构造函数，派生类继承基类的每个构造函数，但有两个例外）：
  1. **默认、拷贝和移动构造函数不被继承**。这些构造函数使用正常规则合成。**继承的构造函数不被视为用户定义的构造函数**。因此，只包含继承构造函数的类将具有合成的默认构造函数
  2. 如果派生类定义的构造函数与基类中的构造函数具有相同的参数，则**该构造函数不被继承**。派生类中定义的那个将替换继承的构造函数

## 🔑 核心要点总结

1. **虚析构函数是基类的标配**：基类通常必须定义虚析构函数以确保通过基类指针删除派生类对象时能正确调用派生类析构函数，避免内存泄漏和未定义行为
2. **虚析构函数阻断合成移动**：定义虚析构函数会阻止编译器合成移动操作，如果需要移动语义，必须显式定义所有五个拷贝控制成员（构造、拷贝构造、移动构造、拷贝赋值、移动赋值、析构）
3. **派生类拷贝控制必须显式处理基类部分**：在定义派生类的拷贝/移动构造函数或赋值运算符时，必须通过初始化列表或显式调用基类运算符来处理基类部分，否则基类部分会被默认初始化而非拷贝/移动
4. **构造/析构期间虚函数绑定静态**：在基类构造函数或析构函数执行期间，对象的类型被视为基类类型，此时调用虚函数不会绑定到派生类版本，以防止访问未初始化的派生类成员
5. **继承构造函数的使用限制**：使用`using`声明可以继承基类构造函数，但不能继承默认/拷贝/移动构造函数，且派生类自定义的构造函数会覆盖同参数列表的继承构造函数

## 📌 考试速记版

* **虚析构函数三原则**：
  - 基类必须有虚析构函数（用于多态删除）
  - 定义了虚析构函数 → 不会合成移动操作（需显式定义五者）
  - 析构函数是虚的 → 派生类析构函数自动为虚

* **危险操作警告**：
  - `delete`基类指针指向派生类对象 + 基类析构函数非虚 = **未定义行为(UB)**

* **派生类拷贝控制模板**：
  ```cpp
  D(const D& d) : Base(d) { /* 处理D成员 */ }  // 拷贝构造
  D(D&& d) : Base(std::move(d)) { /* ... */ }   // 移动构造
  D& operator=(const D& rhs) {
      Base::operator=(rhs);  // 赋值基类部分
      /* 处理D成员 */
      return *this;
  }
  ```

* **构造/析构中的虚函数**：**绝不**在构造/析构函数中调用虚函数（或确保理解其静态绑定行为）

* **继承构造函数速记**：
  - `using Base::Base;` 继承所有构造函数（除默认/拷贝/移动）
  - 默认参数展开为多个构造函数
  - 访问级别、explicit、constexpr属性保留
  - 派生类自定义构造函数优先

**记忆口诀**：
> 基类析构必须虚，删除指针才安全；  
> 移动操作被阻断，显式定义五成员；  
> 派生拷贝莫忘记，基类部分要显式；  
> 构造析构虚函数，静态绑定要小心；  
> using继承构造函数，默认拷贝移动除外。
