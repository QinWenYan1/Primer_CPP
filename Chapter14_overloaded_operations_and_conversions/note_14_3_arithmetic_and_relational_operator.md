## ✅ 知识点1: 算术运算符的设计原则与实现

**理论**
* **算术和关系运算符通常定义为非成员函数**，以便允许左操作数或右操作数的隐式类型转换（如从其他类型转换为类类型）。这些运算符不应改变操作数的状态，因此参数通常定义为**指向const的引用**。
* **算术运算符的行为**：它们生成一个**全新的值**，该值是计算两个操作数的结果。这个新值与任一操作数都不同，通常在函数内部的局部变量中计算得出，并返回该局部变量的副本。
* **与复合赋值运算符的关系**：如果一个类定义了算术运算符（如`+`），通常也应当定义对应的复合赋值运算符（如`+=`）。当两者都存在时，更高效的做法是**用复合赋值运算符来实现算术运算符**，这样只需一次拷贝，然后直接修改副本。

**教材示例代码**
```cpp
// assumes that both objects refer to the same book
Sales_data
operator+(const Sales_data & lhs, const Sales_data & rhs)
{
    Sales_data sum = lhs; // copy data members from lhs into sum
    sum += rhs;
    return sum;
}
```
该定义与之前实现的`add`函数本质相同：首先拷贝左操作数到局部变量`sum`，然后通过复合赋值运算符`+=`将右操作数添加到`sum`中，最后返回`sum`的副本。

**注意点**
* 💡 **实现技巧**：如果一个类同时定义了算术运算符和相应的复合赋值运算符，**应当使用复合赋值来实现算术运算符**。这通常更高效，因为复合赋值直接修改左值，而算术运算符必须返回新对象，利用复合赋值可以避免重复编写计算逻辑，同时利用拷贝构造函数的优化。
* 📋 **术语提醒**：算术运算符(arithmetic operator)、复合赋值运算符(compound assignment operator)、非成员函数(nonmember function)、const引用(reference to const)。

## ✅ 知识点2: 相等运算符的设计原则

**理论**
* **相等运算符的常见定义**：C++中的类通常定义相等运算符(`==`)来测试两个对象是否**等价**。通常的实现是比较每一个数据成员，当且仅当所有对应成员都相等时，两个对象才被认为相等。对于`Sales_data`，应比较`bookNo`（ISBN）、`units_sold`和`revenue`。

**教材示例代码**
```cpp
bool operator==(const Sales_data & lhs, const Sales_data & rhs)
{
    return lhs.isbn() == rhs.isbn() && 
    lhs.units_sold == rhs.units_sold &&
    lhs.revenue == rhs.revenue;
}
bool operator!=(const Sales_data & lhs, const Sales_data & rhs)
{
    return !(lhs == rhs);
}
```

* **设计原则（体现于这些函数）**：
  * 如果一个类需要提供判断两个对象是否相等的操作，应使用`operator==`而不是自定义的命名函数。原因：
    - 用户已经习惯用`==`比较对象，学习成本低；
    - 使用`==`的类可以更容易地与标准库容器和算法配合。
  * 如果定义了`operator==`，它通常应判断两个对象是否包含**等价的数据**（即所有成员对应相等）。
  * 相等运算符应具有**传递性**：如果`a == b`和`b == c`均为真，则`a == c`也应成立。
  * 如果定义了`operator==`，也必须定义`operator!=`。用户期望`==`和`!=`同时可用。
  * 两个运算符中的一个应**委托工作给另一个**：即一个执行实际的比较，另一个调用它（如上述代码中`!=`通过调用`==`并取反实现）。
* **进一步说明**：对于逻辑上应当支持相等比较的类，**通常应该定义`operator==`**，这使该类更容易与标准库算法（如查找、排序）一起使用。

**注意点**
* ⚠️ **一致性要求**：相等运算符的定义必须符合逻辑，且满足传递性，否则可能导致容器或算法行为异常。
* 📋 **术语提醒**：相等运算符(equality operator)、不等运算符(inequality operator)、可传递性(transitive)、等价(equivalent)。

## ✅ 知识点3: 关系运算符的设计原则与Sales_data的决策

**理论**
* **关系运算符的存在意义**：定义了相等运算符的类，往往也定义关系运算符（特别是`<`），因为关联容器（如`set`、`map`）和一些算法依赖**小于运算符**来对元素进行排序。
* **关系运算符应满足的两个要求**：
  1. 定义一种**排序关系**，该关系必须符合用作关联容器键的要求（即严格弱序，见§11.2.2）；
  2. 如果类同时定义了`==`，则关系运算符应与`==`**保持一致**：若两个对象不相等（`!=`为真），则其中一个必须小于另一个（`a < b`或`b < a`成立）。
* **Sales_data为什么不应当定义`<`**：
  * 可能想到用`compareIsbn`的方式，只比较ISBN成员。这虽然满足要求1（可作键），但与`==`不一致，因为`==`还要求`units_sold`和`revenue`相等。此时两个具有相同ISBN但销量或收入不同的对象，用`<`比较时可能互不小于，但`==`却返回假，破坏了要求2。
  * 也可能想到依次比较所有成员：先ISBN，再units_sold，再revenue。但这样的顺序没有**单一逻辑含义**：不同的使用场景可能希望按不同优先级排序（如按销量、按收入），没有一种顺序是天然正确的。
  * 因此，**对于`Sales_data`类，不存在唯一合理的`<`定义**，最好的选择是**不定义`<`**。

**注意点**
* 💡 **最佳实践**：如果类存在**单一逻辑定义**的小于关系，通常应该定义`<`运算符。但如果类也定义了`==`，则必须确保`<`和`==`的定义**产生一致的结果**（即满足要求2），否则不应定义`<`。
* ⚠️ **警告注意**：不要为了迎合关联容器而随意定义`<`，错误的`<`定义可能导致容器元素无序、查找失败或程序逻辑错误。
* 📋 **术语提醒**：关系运算符(relational operator)、小于运算符(less-than operator)、严格弱序(strict weak ordering)、一致(consistent)。

## 🔑 核心要点总结
1. **算术运算符通常定义为非成员函数**，参数为const引用，返回新对象。若同时有复合赋值运算符，应使用复合赋值实现算术运算符以提高效率。
2. **相等运算符应比较所有数据成员**，满足传递性，并同时提供`==`和`!=`，且两者委托实现。定义`==`的类能更好地融入标准库。
3. **关系运算符（特别是`<`）只有在存在单一逻辑定义且与`==`一致时才应定义**。对于`Sales_data`，不存在唯一合理的`<`，故不应定义。
4. 无论定义何种运算符，都需考虑与已有运算符的**一致性**以及**用户预期**，避免产生混淆或不一致的逻辑。

## 📌 考试速记版
* **算术运算符实现套路**：
  ```cpp
  T operator+(const T& lhs, const T& rhs) {
      T sum = lhs;   // 拷贝左操作数
      sum += rhs;    // 调用复合赋值
      return sum;    // 返回副本
  }
  ```
* **相等运算符设计原则**：
  - 定义`==`和`!=`，委托实现
  - 比较所有数据成员，保证传递性
  - 使用户能与标准库无缝配合
* **关系运算符决策准则**：
  - 存在单一逻辑顺序且与`==`一致 → 定义`<`
  - 否则 → 不定义`<`

**记忆口诀**：  
算术复合一块写，相等委托不分裂；  
关系一致才定义，无单逻辑宁缺勿越。
