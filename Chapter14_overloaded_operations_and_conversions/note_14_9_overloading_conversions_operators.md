## ✅ 知识点1: 类类型转换概述

**理论**

## ✅ 知识点3: 转换运算符的定义和语法

**理论**
* **转换运算符(conversion operator)**是一种特殊的成员函数，它将一个类类型的值转换为其他类型的值。
* 一般形式：`operator type() const;`，其中`type`表示目标类型。
* 转换运算符可以定义为任何能作为函数返回类型的类型（除了`void`），但不允许转换为数组类型或函数类型。允许转换为指针类型（包括数据指针和函数指针）以及引用类型。
* 转换运算符**没有显式声明的返回类型**，**没有参数**，并且必须定义为**成员函数**。
* 转换操作通常不应改变被转换的对象，因此通常应定义为`const`成员。

**教材示例代码**
(无)

**注意点**
* ⚠️ 转换运算符不能有参数，不能指定返回类型，必须是成员函数。
* 📋 语法：`operator type() const;`

<a id="id5"></a>
## ✅ 知识点5: 定义带有转换运算符的类(SmallInt)

**理论**
* 作为示例，定义一个表示0~255范围内整数的类`SmallInt`。
* 该类定义了**双向转换**：构造函数将算术类型值转换为`SmallInt`，转换运算符将`SmallInt`对象转换为`int`。

**教材示例代码**
```cpp
class SmallInt {
public:
    SmallInt(int i = 0): val(i)
    {
        if (i < 0 || i > 255)
            throw std::out_of_range("Bad SmallInt value");
    }
    operator int() const { return val; }
private:
    std::size_t val;
};
```

**注意点**
* 💡 构造函数是`nonexplicit`的，允许从`int`隐式转换。
* 📋 转换运算符`operator int()`返回`val`，允许`SmallInt`隐式转换为`int`。

---

<a id="id6"></a>
## ✅ 知识点6: 隐式转换序列（可结合标准转换）

**理论**
* 虽然编译器一次只应用一个用户定义转换，但用户定义转换可以**前面或后面跟着标准(内置)转换**。
* 因此，可以将任何算术类型传递给`SmallInt`构造函数；也可以将`SmallInt`转换为`int`，再将结果`int`转换为其他算术类型。

**教材示例代码**
```cpp
SmallInt si;
si = 4;               // 隐式将4转换为SmallInt，然后调用SmallInt::operator=
si + 3;               // 隐式将si转换为int，然后执行整数加法

// double实参通过内置转换转换为int
SmallInt si = 3.14;   // 调用SmallInt(int)构造函数
// SmallInt转换运算符将si转换为int；
si + 3.14;            // 该int再通过内置转换转换为double
```

**注意点**
* 🔄 用户定义转换前后可以跟随标准转换，从而扩展了可用性。

<a id="id7"></a>
## ✅ 知识点7: 转换运算符的限制（无参数、无返回类型）

**理论**
* 因为转换运算符是隐式应用的，无法传递参数给它们，所以转换运算符不能定义参数。
* 虽然转换函数不指定返回类型，但每个转换函数必须返回其对应类型的值。
* 下面的代码展示了错误示例：

**教材示例代码**
```cpp
class SmallInt;
operator int(SmallInt&);               // 错误：不能是非成员函数
class SmallInt {
public:
    int operator int() const;           // 错误：返回类型是int，但转换运算符不能指定返回类型
    operator int(int c = 0) const;       // 错误：不能有参数
    operator int*() const { return 42; } // 错误：不能有参数列表（这里注释可能误解，但意图是禁止参数）
};
```

**注意点**
* ⚠️ 转换运算符必须是成员函数，不能有返回类型声明，不能有参数。

<a id="id8"></a>
## ✅ 知识点8: Caution: 避免过度使用转换函数

**理论**
* 和重载运算符一样，合理使用转换运算符可以简化类设计和使用。但某些转换可能会误导。
* 当类类型和转换类型之间没有明显的单一映射关系时，转换运算符会产生误导。
* 例如，考虑一个表示日期的`Date`类。提供到`int`的转换可能返回多种含义：如“19800730”表示1989年7月30日，或者返回从某个纪元开始的天数。问题在于没有单一的、一对一的映射，因此更好的做法是不定义转换运算符，而是定义几个普通成员函数来提取不同形式的信息。

**教材示例代码**
(无)

**注意点**
* ⚠️ 如果不存在自然的单一映射，就不要定义转换运算符，而应提供命名成员函数。

<a id="id9"></a>
## ✅ 知识点9: 转换到bool可能导致的意外

**理论**
* 实践中，类很少提供转换运算符，因为用户更可能被自动转换吓到而不是受益。但有一个重要例外：转换为`bool`并不少见。
* 在旧标准中，希望定义到`bool`转换的类面临问题：因为`bool`是算术类型，转换为`bool`的类对象可以在任何需要算术类型的上下文中使用，可能导致意外。
* 例如，如果`istream`有到`bool`的转换，下面代码会编译：
  ```cpp
  int i = 42;
  cin << i;  // 如果到bool的转换不是explicit，这段代码会合法！
  ```
  这段代码本意是输出，但实际会尝试将`cin`转换为`bool`，然后将`bool`提升为`int`（1或0），再执行内置的左移操作，将1或0左移42位。这几乎肯定是错误。

**教材示例代码**
```cpp
int i = 42;
cin << i; // this code would be legal if the conversion to bool were not explicit!
```

**注意点**
* 💡 转换为`bool`需要特别小心，否则可能导致意想不到的行为。

---

<a id="id10"></a>
## ✅ 知识点10: explicit转换运算符

**理论**
* 为了防止上述问题，新标准引入了**显式转换运算符(explicit conversion operators)**。
* 声明为`explicit`的转换运算符只能用于显式类型转换（如`static_cast`），不能用于隐式转换。
* 示例：将`SmallInt`的`operator int`声明为`explicit`。

**教材示例代码**
```cpp
class SmallInt {
public:
    explicit operator int() const { return val; }
    // 其他成员...
};
```

**注意点**
* ⚠️ `explicit`转换运算符可以避免隐式转换带来的意外。
* 📋 即使声明为`explicit`，在特定上下文中（如条件表达式）仍可能隐式应用（C++11后条件表达式允许`explicit`转换为`bool`），但一般要求显式转换。

## ✅ 知识点12: 引入：确保只有一种转换方式

**理论**
* 如果一个类有一个或多个转换，重要的是确保从类类型到目标类型只有一种转换方式。如果存在多种转换路径，将很难编写无歧义的代码。
* 有两种方式会产生多条转换路径：
  1. 两个类提供**相互转换(mutual conversions)**：类A定义接受B的转换构造函数，同时类B定义到A的转换运算符。
  2. 定义多个从/到相关类型的转换（如内置算术类型）。一个类通常最多只应定义一个到算术类型的转换（或从算术类型转换）。

<a id="id14"></a>
## ✅ 知识点14: 参数匹配和相互转换(Argument Matching and Mutual Conversions)

**理论**
* 在下面的例子中，我们定义了两个从`B`获得`A`的方式：使用`B`的转换运算符，或使用接受`B`的`A`构造函数。

**教材示例代码**
```cpp
// 通常在两个类之间定义相互转换是不好的
struct B;
struct A {
    A() = default;
    A(const B&);   // 从B转换到A
    // 其他成员
};
struct B {
    operator A() const; // 也从B转换到A
    // 其他成员
};
A f(const A&);
B b;
A a = f(b); // 错误：歧义 f(B::operator A()) 还是 f(A::A(const B&)) ?
```

**注意点**
* 🔄 因为存在两种获得`A`的方式，编译器不知道使用哪个转换，调用`f`是歧义的。

---

<a id="id15"></a>
## ✅ 知识点15: 相互转换的歧义示例及解决

**理论**
* 上述调用是歧义的，因为两个函数（`B::operator A()`和`A::A(const B&)`）一样好。
* 如果想调用，必须显式地调用其中一个：
  ```cpp
  A a1 = f(b.operator A());  // 使用B的转换运算符
  A a2 = f(A(b));             // 使用A的构造函数
  ```
* 注意，不能通过强制类型转换来解决，因为强制转换本身也会有同样的歧义。

**教材示例代码**
```cpp
A a1 = f(b.operator A()); // ok: use B's conversion operator
A a2 = f(A(b));           // ok: use A's constructor
```

**注意点**
* ⚠️ 相互转换导致的歧义必须通过显式调用某一转换来解决。

<a id="id16"></a>
## ✅ 知识点16: 多重转换到内置类型导致的歧义(Ambiguities and Multiple Conversions to Built-in Types)

**理论**
* 当类定义了多个到（或从）本身存在转换关系的类型的转换时，也会产生歧义。
* 最明显的例子是内置算术类型。如果一个类定义了多个从算术类型的构造函数，或多个到算术类型的转换运算符，就容易产生歧义。

**教材示例代码**
```cpp
struct A {
    A(int = 0);    // 通常定义两个从算术类型的转换是不好的
    A(double);     // 从算术类型的转换
    operator int() const;   // 通常定义两个到算术类型的转换也是不好的
    operator double() const; // 到算术类型的转换
    // 其他成员
};

void f2(long double);
A a;
f2(a); // 错误：歧义 f(A::operator int()) 还是 f(A::operator double()) ?

long lg;
A a2(lg); // 错误：歧义 A::A(int) 还是 A::A(double) ?
```

**注意点**
* 💡 `f2(a)`：两个转换运算符都可以，之后跟随标准转换到`long double`，没有哪一个更好，因此歧义。
* 💡 `A a2(lg)`：`long`到`int`或`double`的转换等级相同（都是标准转换），无法区分。

<a id="id17"></a>
## ✅ 知识点17: 多重算术类型转换示例(A类)及歧义

**理论**
* 在`f2(a)`中，没有转换是`long double`的精确匹配，但任一转换后跟一个标准转换都能达到目标，因此歧义。
* 在`A a2(lg)`中，没有构造函数是`long`的精确匹配，每个都需要先将实参转换后再使用构造函数：`long`→`double`然后`A(double)`，或`long`→`int`然后`A(int)`。这两个转换序列不可区分，因此歧义。
* 当使用用户定义转换时，所需标准转换的等级（如果有）用来选择最佳匹配。例如：
  ```cpp
  short s = 42;
  // 提升short到int 优于 转换short到double
  A a3(s); // 使用 A::A(int)
  ```
  这里提升`short`到`int`优于转换为`double`，所以调用`A::A(int)`。

**教材示例代码**
```cpp
short s = 42;
// promoting short to int is better than converting short to double
A a3(s); // uses A::A(int)
```

**注意点**
* 🔄 当两个用户定义转换都可行时，标准转换的等级决定最佳匹配。

<a id="id19"></a>
## ✅ 知识点19: 重载函数和转换构造函数(Overloaded Functions and Converting Constructors)

**理论**
* 在调用重载函数时，如果两个或多个转换提供了可行匹配，这些转换被认为是同样好的。
* 当重载函数接受的参数类型是不同的类类型，而这些类定义了相同的转换构造函数时，就可能产生歧义。

**教材示例代码**
```cpp
struct C {
    C(int);
    // other members
};

struct D {
    D(int);
    // other members
};

void manip(const C&);
void manip(const D&);
manip(10); // 错误歧义：manip(C(10)) 还是 manip(D(10)) ?
```

**注意点**
* 💡 因为`C`和`D`都有接受`int`的构造函数，`manip(10)`既可以转换为`C`也可以转换为`D`，导致歧义。

## ✅ 知识点20: 重载函数与相同转换构造函数的歧义(C和D)

**理论**
* 调用者可以通过显式构造所需类型来消除歧义：
  ```cpp
  manip(C(10)); // ok: 调用 manip(const C&)
  ```
* **Warning**：在调用重载函数时需要显式使用构造函数或强制转换来转换实参，通常是设计不佳的征兆。

**教材示例代码**
```cpp
manip(C(10)); // ok: calls manip(const C&)
```

<a id="id22"></a>
## ✅ 知识点22: Caution: 转换与运算符的设计准则

**理论**
* 正确设计类的重载运算符、转换构造函数和转换函数需要谨慎。如果类同时定义了转换运算符和重载运算符，很容易产生歧义。
* 一些经验法则：
  - **不要定义相互转换的类**：如果类`Foo`有接受`Bar`对象的构造函数，就不要在`Bar`中定义到`Foo`的转换运算符。
  - **避免转换到内置算术类型**。如果确实需要定义到算术类型的转换，那么：
    - 不要定义接受算术类型的重载运算符版本。如果需要使用这些运算符，转换操作会将类对象转换，然后使用内置运算符。
    - 不要定义到多个算术类型的转换。让标准转换提供到其他算术类型的转换。
  - **最简单的规则**：除了显式的`bool`转换外，避免定义转换函数，并将非显式构造函数限制为“明显正确”的那些。

(无)

**注意点**
* ⚠️ 这些准则有助于避免歧义和意外行为。

<a id="id23"></a>
## ✅ 知识点23: 重载函数和用户定义转换(Overloaded Functions and User-Defined Conversion)

**理论**
* 在调用重载函数时，如果两个（或多个）用户定义转换提供了可行匹配，这些转换被视为同样好。此时不考虑可能需要的任何标准转换的等级。只有当重载集可以使用**相同的转换函数**匹配时，才会考虑是否还需要内置转换。

**教材示例代码**
```cpp
struct E {
    E(double);
    // other members
};

void manip2(const C&);
void manip2(const E&);

// 错误歧义：可以使用两种不同的用户定义转换
manip2(10); // manip2(C(10)) 或 manip2(E(double(10)))
```

**注意点**
* 💡 `manip2(10)`：`C`有从`int`的转换（精确匹配），`E`有从`double`的转换（需要`int`→`double`标准转换）。尽管一个需要标准转换另一个不需要，但因为使用了不同的用户定义转换，调用仍然是歧义的。

---

<a id="id24"></a>
## ✅ 知识点24: 不同用户定义转换导致歧义(E类示例)

**理论**
* 上述例子中，`manip2`的两个可行函数需要不同的用户定义转换，因此调用是歧义的。即使其中一个需要标准转换而另一个不需要，编译器仍会标记为错误。

---

<a id="id25"></a>
## ✅ 知识点25: Note: 只有相同用户定义转换时才考虑后续标准转换

**理论**
* **Note**：在重载函数调用中，附加标准转换的等级仅在可行函数需要相同的用户定义转换时才有意义。如果需要不同的用户定义转换，则调用是歧义的。

## ✅ 知识点27: 重载运算符的候选函数包括成员和非成员

**理论**
* 重载运算符是重载函数。决定对给定表达式应用哪个运算符（内置的还是重载的）使用常规函数匹配。
* 但是，当在表达式中使用运算符函数时，候选函数集比普通函数调用更广。如果`a`有类类型，表达式`a sym b`可能解释为：
  - `a.operatorsym(b)`（成员函数）
  - `operatorsym(a, b)`（非成员函数）
* 与普通函数调用不同，我们不能通过调用形式来区分是调用成员函数还是非成员函数。

<a id="id29"></a>
## ✅ 知识点29: 示例：SmallInt的加法运算符导致歧义

**理论**
* 为`SmallInt`类定义一个加法运算符（友元函数）。
* 该类既有从`int`的转换构造函数，也有到`int`的转换运算符。

**教材示例代码**
```cpp
class SmallInt {
    friend
    SmallInt operator+(const SmallInt&, const SmallInt&);
public:
    SmallInt(int = 0);    // conversion from int
    operator int() const { return val; } // conversion to int
private:
    std::size_t val;
};

SmallInt s1, s2;
SmallInt s3 = s1 + s2;  // 使用重载的operator+
int i = s3 + 0;         // 错误：歧义
```

**注意点**
* 💡 `s1 + s2` 使用重载的`+`（两个`SmallInt`）。
* 💡 `s3 + 0` 有两种可能：将`0`转换为`SmallInt`然后使用重载`+`，或将`s3`转换为`int`然后使用内置`int`加法。两者一样好，导致歧义。

## ✅ 知识点30: Warning: 同时提供转换函数和重载运算符可能导致歧义

**理论**
* **Warning**：为同一个类类型同时提供到算术类型的转换函数和重载运算符，可能导致重载运算符和内置运算符之间的歧义。

## 🔑 核心要点总结
1. **类类型转换**包括转换构造函数（从其他类型到类类型）和转换运算符（从类类型到其他类型）。它们都是用户定义转换。
2. **转换运算符**必须是成员函数，无返回类型，无参数，通常为`const`。可以定义`explicit`转换运算符防止隐式转换。
3. **避免歧义**：不要定义相互转换的类，尽量避免定义到多个算术类型的转换；在重载函数中，如果两个可行函数需要不同的用户定义转换，调用是歧义的，即使其中一个有标准转换另一个没有。
4. **重载运算符的函数匹配**同时考虑成员和非成员版本，并且如果类同时提供转换函数和重载运算符，可能引起歧义。
5. **设计准则**：除了显式的`bool`转换，通常避免定义转换函数；非显式构造函数应限于“明显正确”的情形。

## 📌 考试速记版
* **转换运算符语法**：`operator type() const;`
* **转换构造函数**：`nonexplicit`单参数构造函数定义隐式转换。
* **相互转换歧义**：类A有A(B)构造函数，类B有operator A()，调用f(A)时f(b)歧义。
* **多重算术转换歧义**：类定义多个到算术类型的转换或从算术类型的构造函数时，可能歧义。
* **重载函数歧义**：如果两个重载版本需要不同的用户定义转换，则调用歧义。
* **重载运算符歧义**：如果类既有到算术类型的转换，又有接受该算术类型的重载运算符，可能歧义。
* **经验法则**：避免定义转换运算符（除了explicit bool），避免相互转换，避免多个算术转换。

**记忆口诀**：转换构造和运算符，成员无参无返回；避免相互多算转，重载匹配看异同；显式布尔是例外，设计准则要记清。
